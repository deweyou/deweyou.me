工具调用效果常先卡在 schema 和字段说明，而不是模型“不会用工具”。

模型决定是否调用工具时，会把工具名、参数名和描述当作可读提示。描述越含糊，模型越难判断何时调用、该传什么、何时停止；把输入、输出和失败条件写清，常比继续堆 system prompt 更有效。

Agent 的稳定性很大程度取决于接口设计质量。好的 schema 能减少误调用、空调用和多轮澄清，直接影响延迟、成本和成功率。

把搜索工具参数从 query 改成 user_question，并注明“传入完整自然语言问题，不要只给关键词”，模型通常会比只改提示词更稳定。

这不等于描述越长越好，过长会稀释关键信号；schema 也不能替代权限校验和回退逻辑。

如果把 **工具描述是隐形提示词** 当成检查题，你的系统里哪一段还只是靠默认假设在运转，而没有被设计成可观测、可验证、可回退的工程机制？
