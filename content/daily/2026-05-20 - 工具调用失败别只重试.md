title: 工具调用失败别只重试
date: 2026-05-20

工具调用失败时，单纯重复请求是最差的恢复策略；正确做法是按失败类型分类，匹配不同的恢复路径。

LLM 的工具调用会产出多种失败——参数错误、函数名不存在、格式非法、权限不足、超时等。很多 Agent 框架对所有失败做统一重试，这会浪费 token 并可能反复触发同一个错误。

真正影响 Agent 可靠性的不是单次调用成功率，而是工具链路的恢复机制。分类处理能把偶然失败（网络抖动、限流）与必然失败（参数语义错误、权限缺失）分开——前者可安全重试，后者必须修正输入或降级。

一个查询天气的 Agent 收到 "city not found"。重试 3 次用同样参数只会浪费 token。正确做法：识别为参数语义错误，把错误信息注入下一轮提示词，让模型换方式描述城市名，或回退到模糊匹配。

① 每次重试都消耗 token 并累积到上下文，多次失败会让后续对话越来越慢且越来越贵；② 不是所有错误信息都适合直接给模型看——有些错误码会误导模型产生方向错误的"修正"，反而越改越离谱。

如果把 **工具调用失败别只重试** 当成检查题，你的系统里哪一段还只是靠默认假设在运转，而没有被设计成可观测、可验证、可回退的工程机制？
