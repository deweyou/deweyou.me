Agent 的沙箱边界不能靠“路径看起来安全”来推断，应该变成显式能力授权。OpenAI Agents Python 5 月的 v0.17.0 / v0.17.1 连续修了 sandbox local source materialization、archive extraction、git repo subpath validation 这类问题，信号很清楚：Agent 一旦能读本地文件、解包归档、拉代码、执行工具，路径就是权限系统的一部分。

一个常见误区是把 sandbox 当成“临时工作目录”。真正的风险在于材料进入沙箱之前：LocalDir 指向哪里、绝对路径是否越过 base_dir、压缩包里是否有意外结构、git subpath 是否能绕到仓库外。OpenAI Agents SDK v0.17.0 的迁移方式是：默认只允许 base_dir 内的 local source，外部可信目录必须通过 Manifest.extra_path_grants / SandboxPathGrant 明确声明，最好还是 read-only。

这对 Agent Engineer 很实用：不要只在 tool executor 里做 allowlist，还要给“上下文材料注入”建模。比如 OPC 这类长期运行的 Agent 平台，如果允许用户把项目目录、知识库、凭据挂载进任务环境，就应该把每个挂载看作一张 capability：来源、目标、只读/可写、生命周期、审计 ID 都要记录。否则一次“帮我读一下这个目录”的便利，很容易变成隐式越权。

可以从一个小改动开始：下次设计文件工具或沙箱任务时，先问“这个 Agent 需要哪几条最小路径能力”，而不是问“这个目录能不能访问”？
