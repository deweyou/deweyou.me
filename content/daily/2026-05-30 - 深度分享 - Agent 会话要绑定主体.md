## 一句话结论

Agent 平台的可靠边界不该只放在 prompt 或工具列表里，而要把“谁在运行、会话属于谁、任务能访问什么”做成运行时协议的一部分。

## 背景

这次补充的新点是：本周几个 Agent 生态信号都指向同一个方向——Agent 正在从“本地脚本调用工具”变成“可托管、可授权、可审计的运行时”。

MCP 2026-07-28 RC 在 5 月 29 日发布，明确提醒实现方要通过版本协商处理协议演进。

同日 MCP Python SDK v1.27.2 加入 AccessToken 的 subject/claims、把 transport session 绑定到 authenticated principal，并把 experimental tasks 限定在创建它的 session。

OpenAI Agents Python 5 月下旬的更新也集中在 MCP SSE 传输加固、缺失工具恢复、trace 类型导出等运行时问题；Anthropic cookbook 则新增了 Agent SDK 在 Docker、Modal、Kubernetes 上托管的示例。

这些不是“又多了几个 API”，而是在提醒：真正难的是 Agent 被长期运行、多人使用、跨工具访问后，权限和状态会不会串。

## 核心机制

1. 主体绑定比“会话 ID”更重要。

   - 会话 ID 只能说明一次连接或一次任务是谁创建的，但 authenticated principal 才说明它代表哪个用户、租户或服务账号。
   - 工具调用、缓存、长任务、回调都应该继承这个主体，而不是只继承 prompt 里的自然语言身份描述。

2. 传输连接不是安全边界，运行时状态才是。

   - SSE、HTTP stream、WebSocket 这类长连接很容易让工程师误以为“连上了就是同一个上下文”。
   - 但一旦重连、代理、后台任务或异步回调介入，如果没有把 session 与主体、scope、claims 绑定，任务就可能拿到不属于自己的工具结果或缓存。

3. 工具权限要按 action 和资源拆开。

   - Agent 看到一个 `github` 或 `database` 工具，不等于它应该拥有所有仓库、所有表、所有写权限。
   - 更稳的做法是把权限表达为 capability：能读哪个 repo、能写哪个环境、能否发外部消息、是否需要审批。模型负责选择动作，运行时负责强制边界。

4. 长任务必须有 ownership。

   - Agent 工程里常见的“后台跑评测、继续爬数据、稍后回调”本质上都是跨轮次任务。
   - 它们必须记录 created_by、session_id、scope、输入摘要、可访问资源和过期策略。否则任务完成时，系统很难判断结果应该回到哪个 thread、能不能被当前用户查看、失败后能不能自动重试。

5. 协议版本要进入兼容性设计。

   - MCP RC 的版本协商提醒很关键：Agent 平台会同时面对新旧 client/server。
   - OPC 这类工程系统不能假设所有工具端同步升级，而要在连接建立时记录协议版本、能力清单和降级路径。

## 常见误区

误区一：把权限写进 system prompt 就够了。Prompt 可以影响模型选择，但不能阻止工具执行。真正的权限必须在工具网关、运行时调度器、凭证下发和审计日志里强制执行。

误区二：只要每个用户一个 thread 就不会串。Discord/Slack thread 只是交互容器，不是完整的安全上下文。后台任务、缓存、MCP session、数据库连接池、文件系统临时目录都可能绕开 thread 边界。

误区三：先把 Agent 跑起来，权限以后再补。权限、会话和任务 ownership 属于底层数据模型；后补通常会遇到历史任务无法归属、日志不可审计、缓存无法区分租户的问题。

## 工程落地

如果把 OPC 里的 Agent 看成“能调用内部工具的自动化同事”，那第一版架构就应该有一个最小运行时账本：`principal_id`、`session_id`、`task_id`、`capabilities`、`tool_version`、`protocol_version`、`approval_state`、`trace_id`。

每次工具调用都从账本派生临时凭证，而不是让模型直接持有长期 token。

实现上可以先做三层边界：第一层是入口鉴权，确认请求来自哪个用户/服务；第二层是 capability resolver，把用户、项目、环境映射成可调用工具与资源。

第三层是 tool gateway，在执行前校验 action/resource/scope，并把结果写入 trace。对于 coding agent，尤其要区分读代码、改代码、开 PR、触发 CI、部署这五类权限。

对于数据/运营自动化，要区分读、导出、写回、通知外部系统。

判断一个 Agent 框架是否适合 OPC，不只看模型效果，还要看它是否支持会话恢复、工具缺失恢复、trace 导出、MCP/工具传输加固、后台任务归属和版本协商。

没有这些能力，demo 可能很顺，但一旦多人、多项目、长任务并发，就会变成难排查的“状态串线”问题。
