Agent 平台的“能力”不应该只是提示词里的一段描述，也不应该只是运行时临时塞进去的一组 tools；它更像一个可以被命名、组合、授权、评测和回放的一等对象。

Pydantic AI V2 在 6 月 23 日稳定发布时，把 capabilities 放到核心位置：一个 capability 可以把工具、hooks、instructions 和 model settings 打包成同一个可组合单元。这个变化的工程意义不是“多了一个抽象”，而是把过去散落在 agent 初始化、tool registry、middleware 和 prompt 里的能力边界收束到一个可管理对象上。

对 OPC 这种多 Agent / 多工作流平台来说，这会影响三件事。第一，能力目录可以按任务动态挂载，而不是所有工具一开始都暴露给模型。第二，权限审批可以绑定 capability，而不只是绑定单个 tool name。第三，评测和回放可以回答“这次失败是模型问题、工具问题，还是 capability 组合问题”。

常见误区是把 capability 理解成“工具包”。工具包只解决调用入口，能力单元还应该包含前置约束、上下文注入、观测字段、失败恢复和可撤销边界。否则工具越多，Agent 反而越难稳定。

如果我们给 OPC 设计能力目录，哪些能力应该默认隐藏，只有在任务风险、用户授权或上下文证据满足时才显露出来？
