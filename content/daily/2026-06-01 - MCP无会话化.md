MCP 最新草案里最值得 Agent Engineer 关注的变化，不是又加了某个工具字段，而是它在把协议从“连接里藏状态”推向“每次请求都自描述”。2026-07-28 RC 已在 5 月 29 日发布，草案明确移除 protocol-level sessions、`Mcp-Session-Id` 和 `initialize/initialized` 握手，要求请求在 `_meta` 里携带协议版本、客户端身份和能力。

这会改变工具平台的工程边界：以前很多实现会默认“同一条连接 = 同一个会话 = 可以记住上下文”。新方向更像 HTTP/JSON-RPC 的可复放请求：版本协商用 `server/discover`，跨调用状态要变成服务端显式铸造的 handle，再作为普通 tool argument 传回去。

对 Agent 系统来说，这其实是好事。无状态协议更容易做负载均衡、重试、录制回放、审计和缓存；也更容易把 tool list、resource list 这类发现结果缓存起来。草案甚至要求 list 类结果带 `ttlMs` 和 `cacheScope`，并建议工具列表稳定排序，以提高客户端缓存和 LLM prompt cache 命中率。

代价是：你不能再把“记忆、权限、事务进度”偷偷塞进连接生命周期。OPC 或内部 Agent 平台如果接 MCP，应该尽早把状态模型拆成三层：协议请求自描述，业务状态用 handle 显式引用，长期记忆另走受控存储。否则未来迁移到新版 MCP 时，最容易坏的不是工具 schema，而是那些没人写进接口契约的隐式会话假设。

如果今天要重构一个工具服务器，你会把哪些状态留在请求参数里，哪些状态变成短期 handle，哪些状态必须进入可审计的长期记忆？
